# Domain Pitfalls

**Domain:** 面向开发者的本地桌面剪贴板管理器（brownfield）
**Researched:** 2026-03-27
**Research focus:** 桌面监听、结构化检测、预览正确性、模糊搜索质量、测试策略
**Overall confidence:** HIGH

## Why This Matters For This Project

这个项目不是从零做一个“能记住文本”的剪贴板工具，而是在已有 Tauri + React + Rust + SQLite 桌面客户端上，把“开发者内容理解与回查”做深。这里的失败模式和通用笔记应用不同：真正会拖垮体验的，不是功能少，而是监听不可信、结构化识别飘移、详情预览不一致、搜索找不到、以及桌面端打包后行为和测试环境不一致。

从当前代码库分析看，这些风险已经有前兆：监听生命周期不可控、预览职责分散、搜索仍依赖 `%LIKE%`、测试不够 hermetic、存储路径存在 brownfield 漂移。因此 roadmap 不应该先加更多“可展示的功能”，而应该先消除会导致错误记录、错误展示、错误检索的底层坑。

## Recommended Phase Buckets

| Phase   | Goal                         | Why it must exist                                                         |
| ------- | ---------------------------- | ------------------------------------------------------------------------- |
| Phase 1 | 监听生命周期与桌面可靠性加固 | 先确保“开始/停止监听”和打包后桌面行为可靠，否则后续优化都建立在错误输入上 |
| Phase 2 | 捕获模型与检测版本化         | 先稳住原始数据和 subtype 判定边界，避免解析规则升级后历史数据失真         |
| Phase 3 | 预览契约与结构化渲染修正     | 让预览只消费可信数据，不再同时承担抓取、解析、回写三种职责                |
| Phase 4 | 搜索索引与相关性             | 开发者搜索不是普通全文检索，必须尽早建立 index 和评估集                   |
| Phase 5 | 可靠性测试与 CI 发布门禁     | 不补上桌面链路测试，前面几阶段都会持续回归                                |
| Phase 6 | 存储统一与迁移收敛           | brownfield 路径、缓存和 schema 漂移会持续污染 Phase 1-5 的结果            |

## Critical Pitfalls

### Pitfall 1: 把监听实现成“可见状态位 + fire-and-forget 任务”

**What goes wrong:**  
团队往往把“是否监听”实现成一个前端状态和一个后台轮询循环，结果 `stop` 只改 UI 状态，不真正取消后台任务；多次 start/stop 后出现重复轮询、重复订阅、重复入库，用户看到的是“我明明停了，为什么还在记？”。

**Why it happens:**  
因为把桌面剪贴板当成普通轮询源处理，忽略了平台差异和服务生命周期。Windows 提供 `AddClipboardFormatListener` / `WM_CLIPBOARDUPDATE` 一类变更通知；Windows 还提供 `GetClipboardSequenceNumber` 追踪变更序号；macOS 则常依赖 `changeCount` 和 pasteboard type 检查。常见错误是把这些平台差异抹平为一个固定定时器。

**Consequences:**  
错误采集、重复记录、电量与 CPU 空转、监听停止失效、自身复制回流、远端剪贴板噪音进入历史，最终让“历史记录”本身不可信。

**Warning signs:**

- 停止监听后，历史列表仍继续新增条目
- 多次启停后，重复条目明显增多
- 空闲时仍稳定出现轮询唤醒或 CPU 抖动
- “忽略下一次复制”之类的状态存在，但实际不生效
- 远端剪贴板或应用自写回内容频繁进入历史

**Prevention:**

- 把监听实现为显式可取消的服务对象，不是丢给 `spawn` 后就不管
- 为每个平台保留独立 adapter：事件通知、变更序号、退避轮询策略分别处理
- 用稳定去重键记录“这次变更”而不是只对内容 hash 去重
- 把“忽略下一次复制”“忽略某些来源应用”“忽略远端剪贴板”收敛成统一策略层
- 为 start/stop/restart 增加集成测试，验证后台任务数、订阅数、状态位一致

**Phase should address it:**  
Phase 1，必要时拆出一个 1.x 子阶段专门处理 monitoring service lifecycle。

**Confidence:** HIGH

### Pitfall 2: 过早把多格式剪贴板压成单一字符串，导致检测漂移无法修复

**What goes wrong:**  
团队会在捕获阶段立刻把内容归一成 `text/plain` 或一个自定义 subtype，然后丢掉原始 type 列表、HTML 片段、来源元数据、二进制表示。短期看实现简单，长期会造成 subtype 漂移无法回放：今天识别错的 JSON、URL、代码、颜色，明天即使修复 detector，也无法对历史记录重新判定。

**Why it happens:**  
因为把“捕获”与“理解”混成一步。实际上桌面剪贴板常同时携带多种表示：macOS 的 pasteboard API 暴露多 type 写入/读取；Windows 还有专门的 HTML Clipboard Format。开发者工具如果只保存首个可读字符串，就等于主动放弃后续纠错能力。

**Consequences:**  
结构化内容误判为普通文本、浏览器复制内容丢上下文、同一条历史在不同版本中 subtype 反复变化、预览逻辑越来越依赖脆弱正则，最终“检测准确率”优化会变成一串不可逆的兼容补丁。

**Warning signs:**

- 同一份内容在不同版本中被识别成不同 subtype
- 浏览器复制的富文本/链接在历史里只剩纯文本
- 想修复 detector 时，必须要求用户重新复制数据
- detector 逻辑高度集中在单个超大模块，规则彼此覆盖
- “unknown type” 直接静默降级成 plain text，没有任何诊断信息

**Prevention:**

- 分离 capture schema 和 derived schema：原始载荷、规范化文本、检测结果分别存
- 持久化 `detector_version`、`parser_version`、`preview_schema_version` 和置信度
- 保留原始 type 列表、来源应用、尺寸/长度等辅助元数据，便于重判
- 提供后台 re-detect / re-index 任务，允许历史记录随新规则迁移
- 建立 golden corpus：JSON、URL、代码块、颜色、文件路径、终端输出、浏览器复制片段都要覆盖

**Phase should address it:**  
Phase 2，且应先于大规模预览改造。

**Confidence:** HIGH

### Pitfall 3: 不尊重 transient / concealed / auto-generated / remote markers

**What goes wrong:**  
本地剪贴板不只承载“用户主动复制”的内容，还会被密码管理器、文本替换工具、宏工具、远端设备同步、剪贴板工具自身回写所利用。如果历史管理器忽略这些 marker，就会把本应跳过、遮罩或短暂存在的数据持久化，轻则噪音爆炸，重则隐私事故。

**Why it happens:**  
很多团队只围绕 MIME/type 做识别，没有把“这个内容是否应该被记录”视为一等问题。社区长期约定的 `org.nspasteboard.TransientType`、`ConcealedType`、`AutoGeneratedType`、`org.nspasteboard.source` 以及 `com.apple.is-remote-clipboard` 等标记，常被遗漏。

**Consequences:**  
密码或敏感文本进入本地历史、文本展开器生成内容污染历史、远端设备同步内容反复入库、来源应用误判、用户对工具丧失信任。

**Warning signs:**

- 密码管理器、替换工具、远端剪贴板内容经常出现在历史中
- “来源应用”频繁显示为本应用本身而不是原始来源
- 用户反馈“某些临时复制不该被记录”
- 忽略规则只能按内容类型做，不能按 pasteboard marker 或来源应用做
- 清理历史后，敏感内容仍可从缓存或磁盘恢复

**Prevention:**

- 建立 capture policy 层，先判断“该不该记”，再做 subtype 检测
- 默认支持 transient / concealed / auto-generated / remote 等 marker
- 对 concealed 类型默认跳过持久化，或至少只保留遮罩态与最小元数据
- 引入 per-app / per-type ignore 规则和 one-shot skip 机制
- 对“来源应用”使用显式 source marker 优先于前台应用推断

**Phase should address it:**  
Phase 1 先补基础忽略策略，Phase 2 再补 marker-aware capture schema。

**Confidence:** HIGH

### Pitfall 4: 让预览层既负责解析、又负责抓取、还负责回写

**What goes wrong:**  
详情页一旦同时承担“解析内容”“远程抓取元数据”“选择最佳 renderer”“补写缓存/状态”，就很容易出现预览与原始条目分叉。典型后果是：列表看起来是 JSON，详情页却按 plain text 渲染；旧请求在用户切换选中项后回写新状态；巨大 JSON 或 URL 预览把 UI 卡死；更糟的是预览逻辑意外修改历史条目。

**Why it happens:**  
因为预览链路经常从“展示层优化”慢慢变成“内容理解中枢”，最终前后端都在做解析、抓取和缓存，职责边界消失。

**Consequences:**  
详情不可信、预览闪烁、卡顿、错误回退、网络访问不可控、同一条记录在不同视图中的语义不一致。

**Warning signs:**

- 同一条记录在列表摘要、详情页、复制回写时展示不一致
- 选中项切换后，旧异步请求仍能覆盖当前预览
- 打开大 JSON / 代码块时 UI 明显掉帧或冻结
- 远程 URL 预览在未明确授权时自动发请求
- 预览缓存 key 只基于 entry id，不包含解析器/renderer 版本

**Prevention:**

- 把 entry 设为不可变事实源，preview 只能是派生数据
- 规定单一 authoritative preview pipeline，避免前后端重复解析
- 预览缓存使用 `content_hash + detector_version + preview_schema_version`
- 所有异步预览任务都要有取消、超时、尺寸预算和降级路径
- 为 JSON、URL、颜色、代码、文件等建立 preview contract tests，重点测 stale response 与 fallback 行为

**Phase should address it:**  
Phase 3，且在此之前不要继续叠加新的 renderer。

**Confidence:** MEDIUM-HIGH  
说明：内部证据很强；外部证据主要来自同类项目 issue 模式，而非统一官方规范。

### Pitfall 5: 用 `%LIKE%` + debounce 假装实现了“开发者搜索”

**What goes wrong:**  
开发者搜索包含路径、URL、query string、颜色值、camelCase、snake_case、命令输出、hash、JSON key 等多种 token 形态。团队常见错误是继续用 `%LIKE%` 扫整表，再手写一点模糊排序。这样一开始“能搜到”，但很快会出现召回差、排序怪、数据量一大就明显变慢。

**Why it happens:**  
因为把搜索当作列表过滤器，而不是独立产品能力。SQLite FTS5 已经提供 prefix index、trigram tokenizer、bm25/rank、highlight/snippet 等能力；如果不在 schema 层提前设计，后面只能不断加 patch。

**Consequences:**  
搜索结果不稳定，`JSON.parse`、文件路径片段、URL 子串、颜色码等高频场景很难命中；随着历史增长，列表加载和搜索相互影响，用户会直接觉得“搜不到”。

**Warning signs:**

- 用户需要不断换关键字顺序才能搜到同一条记录
- `JSON.parse` 搜不到 `parse`，或路径后缀搜不到目标文件
- 结果排序无法解释，最佳命中不在前几位
- 数据量增加后，输入框 debounce 仍能触发明显卡顿
- 搜索评估只看“能不能返回”，没有 recall / ranking 基线

**Prevention:**

- 建立独立搜索索引，不在主表上长期依赖 `%LIKE%`
- 使用 FTS5 external-content table，按字段设置权重
- 针对开发者内容设计 tokenizer：至少考虑 prefix，必要时引入 trigram 支持子串匹配
- 用 `bm25` / `rank` 做排序，并保留 highlight/snippet 让结果可解释
- 维护搜索评估集：路径、URL、JSON、代码符号、颜色、终端输出都要有固定 query expectation

**Phase should address it:**  
Phase 4。

**Confidence:** HIGH

### Pitfall 6: 测试只覆盖 UI 片段，不覆盖“真实桌面链路 + brownfield 迁移”

**What goes wrong:**  
很多桌面剪贴板项目会积累不少组件测试和 store 测试，但真正容易出事故的是监听启停、真实文件路径、缓存清理、预览竞态、打包后的快捷键/权限、数据库迁移。这些如果不做 hermetic integration coverage，测试再多也会给出“假绿”。

**Why it happens:**  
因为桌面端测试成本高，团队会下意识把质量门禁退回到单元测试和 build 成功。但 brownfield 项目最危险的回归，正是“测试环境跑得好，真实用户目录/打包环境才炸”。

**Consequences:**  
停止监听无效、缓存文件泄漏、真实用户目录被测试污染、发布后快捷键/粘贴自动化失效、解析器升级后历史记录回归。

**Warning signs:**

- CI 只构建，不运行 `cargo test` / `vitest` / 打包后 smoke test
- 测试仍读写真实用户目录或默认 app data 路径
- 性能测试全部标记为 ignored，长期没人跑
- 修复 bug 只能靠手工回归，无法写出稳定重现测试
- 发布后才发现 dev mode 和 packaged app 行为不同

**Prevention:**

- 为 clipboard、filesystem、config path、preview fetch 建 injectable adapter
- 所有状态层和处理器都支持 temp root 注入，禁止测试落到真实目录
- 维护结构化 golden corpus 与 end-to-end fixture pipeline
- CI 至少执行：前端测试、Rust 测试、关键平台 smoke test、迁移测试
- 把“打包后行为”列为发布门禁，不再用 dev mode 替代验收

**Phase should address it:**  
Phase 5，但其中“temp root 注入”和“start/stop 集成测试”应在 Phase 1 并行开始。

**Confidence:** HIGH

## Moderate Pitfalls

### Pitfall 7: brownfield 路径、缓存和 app identity 漂移长期共存

**What goes wrong:**  
应用名、配置目录、数据库路径、图片缓存、统计面板、日志路径在历史演进中分裂成多个根目录，功能看似还能跑，但统计不准、迁移隐式、删除不彻底、测试串到真实数据。

**Warning signs:**

- 不同模块指向不同 app data 根目录
- 清空历史后磁盘文件仍残留
- 统计面板显示的数据库大小和真实文件不一致
- 重命名产品后出现“老数据找不到”或“新旧缓存混在一起”

**Prevention:**

- 建立唯一 app-path helper，路径全量收口
- 为每次路径/应用名变更写显式迁移，不依赖“兼容读取”长期混用
- 删除、清空、过期清理都走事务化资源回收
- 把 migration smoke test 纳入发布流程

**Phase should address it:**  
Phase 6；但如果当前路径分裂已影响测试或统计，应提前插入 1.x 修复。

**Confidence:** HIGH

### Pitfall 8: 假设开发环境行为等于打包后的桌面行为

**What goes wrong:**  
快捷键、无障碍权限、托盘菜单、窗口焦点、AppleScript/系统 API、Wayland/Qt 等问题，在 dev mode 和 packaged app 的表现往往不同。团队如果只在开发环境验证，最终会在线上遇到“本地能复现不了、用户机器必现”的桌面特有问题。

**Warning signs:**

- 粘贴回放在开发环境可用，安装包里失败
- 快捷键在密码框、远程桌面、Wayland 或系统升级后异常
- 托盘菜单或浮层只在特定图形平台/窗口管理器下失效
- 问题只能通过手工重装或切换平台后才出现

**Prevention:**

- 为 packaged app 建单独 smoke checklist，不拿 dev mode 代替
- 平台自动化逻辑通过 adapter 封装，避免硬编码进业务流程
- 在设置页暴露权限诊断和当前平台能力状态
- 记录图形平台、窗口系统、权限状态到日志/诊断报告

**Phase should address it:**  
Phase 1 先补诊断与 smoke 路径，Phase 5 再把它自动化进发布门禁。

**Confidence:** MEDIUM-HIGH

## Minor Pitfalls

### Pitfall 9: 未知内容类型静默降级，没有观测面

**What goes wrong:**  
所有无法识别的内容都直接当作普通文本处理，不记录原始 type、失败原因或 detector confidence，导致团队永远不知道最常见的漏判来自哪里。

**Prevention:**

- 为 unknown / ambiguous case 保留最小诊断数据
- 聚合 detector miss 统计，优先修复高频来源应用和高频 type
- 在详情页明确标识“降级渲染”，不要伪装成已准确识别

**Phase should address it:**  
Phase 2。

**Confidence:** HIGH

## Phase-Specific Warnings

| Phase Topic  | Likely Pitfall                             | Mitigation                                            |
| ------------ | ------------------------------------------ | ----------------------------------------------------- |
| 监听生命周期 | 停止监听只改 UI 状态，后台任务未取消       | 服务对象化 + 可取消任务 + start/stop/restart 集成测试 |
| 桌面可靠性   | 快捷键、托盘、焦点、权限只在 dev mode 验证 | packaged smoke test + 平台诊断面板                    |
| 捕获模型     | 过早归一成字符串，历史数据不可重判         | 原始载荷/规范化文本/检测结果三层分离                  |
| parser 漂移  | detector 升级后历史 subtype 混乱           | detector_version + re-detect job + golden corpus      |
| 预览正确性   | 旧异步请求覆盖新选中项，或预览回写条目     | immutable entry + cancel/timeout + contract tests     |
| URL/远程预览 | 未经授权自动请求外网或内网地址             | 显式用户触发 + 地址策略 + 限时取消                    |
| 搜索质量     | `%LIKE%` 扫整表，排序和命中不可解释        | FTS5 + prefix/trigram + bm25/rank + 评估集            |
| 存储迁移     | app data 根目录分裂，删除不干净            | 单一 path helper + 显式 migration + 事务化清理        |
| 测试体系     | 只有组件测试，没有真实桌面链路测试         | temp root 注入 + 端到端 fixture + CI 质量门禁         |

## Recommended Roadmap Guardrails

- 先修“输入可信度”，再修“展示漂亮度”。监听与 capture schema 不稳时，不要继续扩展 preview feature surface。
- 把 detector、preview、search 都视为 versioned contract，而不是一组临时 heuristics。
- 所有 brownfield 改造都要求迁移策略：路径、缓存、索引、schema、preview cache 一律不能靠“兼容读取”无限期拖延。
- 搜索质量必须有评估集，不能只凭主观感受调排序。
- 任何会接触真实用户目录、权限、托盘、快捷键、焦点、系统 API 的改动，都必须经过打包后 smoke 验证。
